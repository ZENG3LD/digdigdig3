# Lighter.xyz Trading API Specification

Source: https://apidocs.lighter.xyz
Base URL (Mainnet): `https://mainnet.zklighter.elliot.ai`

---

## Architecture Note: No Dedicated REST Order Endpoints

Lighter does NOT use traditional REST endpoints for order placement and cancellation
(e.g., `POST /order`, `DELETE /order`). All write operations (create, cancel, modify
orders) are submitted as **signed transactions** via two generic transaction endpoints:

- `POST /api/v1/sendTx` — single transaction
- `POST /api/v1/sendTxBatch` — up to 50 transactions in one request

Each transaction must be cryptographically signed using an API private key before
submission. The `tx_type` integer field in the body determines what action is performed.

---

## Order Types Supported

From `accountActiveOrders` response enum and SDK constants:

| Order Type | SDK Constant | Description |
|---|---|---|
| Limit | `ORDER_TYPE_LIMIT` | Execute at specified price or better |
| Market | `ORDER_TYPE_MARKET` | Execute immediately at current market price |
| Stop-Loss | `ORDER_TYPE_STOP_LOSS` | Triggered when markPrice <= triggerPrice (sell) |
| Stop-Loss Limit | `ORDER_TYPE_STOP_LOSS_LIMIT` | Stop-loss trigger + limit price execution |
| Take-Profit | `ORDER_TYPE_TAKE_PROFIT` | Triggered when markPrice >= triggerPrice (buy) |
| Take-Profit Limit | `ORDER_TYPE_TAKE_PROFIT_LIMIT` | Take-profit trigger + limit price execution |
| TWAP | `ORDER_TYPE_TWAP` | Time-weighted average price — splits large orders over time |
| TWAP Sub | (internal) | Child sub-orders generated by a parent TWAP order |
| Liquidation | (internal) | System-generated for forced liquidations |

The `type` field in order responses returns one of: `limit`, `market`, `stop-loss`,
`stop-loss-limit`, `take-profit`, `take-profit-limit`, `twap`, `twap-sub`, `liquidation`.

---

## Time-in-Force Options

| TIF | SDK Constant | Description |
|---|---|---|
| Good Till Time | `ORDER_TIME_IN_FORCE_GOOD_TILL_TIME` | Valid until `order_expiry` unix timestamp |
| Immediate or Cancel | `ORDER_TIME_IN_FORCE_IMMEDIATE_OR_CANCEL` | Fill immediately or cancel; never rests on book |
| Post-Only | `ORDER_TIME_IN_FORCE_POST_ONLY` | Maker-only; cancels if it would cross as taker |

The `time_in_force` field in responses returns one of: `good-till-time`,
`immediate-or-cancel`, `post-only`, `Unknown`.

---

## Additional Order Execution Options

| Option | Field | Description |
|---|---|---|
| Reduce-Only | `reduce_only: bool` | Ensures the order moves position closer to zero. Canceled if fill would reverse position. |
| Post-Only | (via time_in_force) | See above |

## Order Placement

### Single Transaction
```
POST https://mainnet.zklighter.elliot.ai/api/v1/sendTx
```

Body must be signed before submission. The SignerClient handles signing via
`sign_create_order()` which produces the `tx_type` and `tx_info` payload.

**Python SDK example:**
```python
tx, tx_hash, err = await client.create_order(
    market_index=0,           # uint8 — market identifier (0 = ETH perps)
    client_order_index=1234,  # int64 — caller-assigned unique ID for reference/cancel
    base_amount=10,           # int — amount in base asset units (10 = 0.001 ETH)
    price=3100_00,            # int — price as integer (3100_00 = $3100.00)
    is_ask=False,             # bool — False = buy, True = sell
    order_type=client.ORDER_TYPE_MARKET,
    time_in_force=client.ORDER_TIME_IN_FORCE_IMMEDIATE_OR_CANCEL
)
```

**Market order convenience wrapper:**
```python
tx, tx_hash, err = await client.create_market_order(...)
```

**WebSocket alternative (sendTx via WS):**
```json
{
  "type": "jsonapi/sendtx",
  "data": {
    "tx_type": <INTEGER>,
    "tx_info": { ... }
  }
}
```

### Batch Order Placement
```
POST https://mainnet.zklighter.elliot.ai/api/v1/sendTxBatch
```

Up to **50 transactions** in one batch request. Each transaction in the batch
consumes one unit of Volume Quota.

Transaction type `L2CreateGroupedOrders` (tx_type=28) allows creating a group of
related orders (e.g., entry + stop-loss + take-profit) that count as a single quota
unit (not 3 separate units).

**WebSocket alternative:**
```json
{
  "type": "jsonapi/sendtxbatch",
  "data": {
    "tx_types": "[INTEGER, INTEGER, ...]",
    "tx_infos": "[tx_info, tx_info, ...]"
  }
}
```

### Key Order Parameters

| Parameter | Type | Description |
|---|---|---|
| `market_index` | uint8 | Market identifier (e.g., 0 = ETH perps) |
| `client_order_index` | int64 | Caller-assigned unique ID for this order (used for cancellation reference) |
| `base_amount` | int | Amount in base asset units as integer |
| `price` | int | Price as integer (precision defined by market's `supported_price_decimals`) |
| `is_ask` | bool | `true` = sell/ask, `false` = buy/bid |
| `order_type` | enum | Order type constant (see table above) |
| `time_in_force` | enum | TIF constant (see table above) |
| `reduce_only` | bool | Optional; restricts to position-reducing fills |
| `trigger_price` | int | Required for stop-loss/take-profit orders |
| `order_expiry` | int64 | Unix timestamp for GTT orders |

## Order Cancellation

### Cancel Single Order

Submitted via `sendTx` with cancel transaction type. Requires `order_index` that
matches the `client_order_index` provided at creation.

**Python SDK:**
```python
tx, tx_hash, err = await client.create_cancel_order(
    order_index=<client_order_index_used_at_creation>
)
```

### Cancel All Orders

**Python SDK:**
```python
await client.cancel_all_orders(...)
```

Cancel-all uses transaction type `L2CancelAllOrders` (tx_type=5). Cancels do NOT
consume Volume Quota.

---

## Order Modification (Amend)

Supported via signed transaction. The SDK provides `sign_modify_order()` / modify
wrappers. This corresponds to transaction type `L2ModifyOrder` (tx_type=17).

Modification consumes 1 Volume Quota unit.

## Query: Get Open Orders

```
GET https://mainnet.zklighter.elliot.ai/api/v1/accountActiveOrders
```

**Parameters:**

| Name | Type | Required | Description |
|---|---|---|---|
| `account_index` | int64 | Yes | Account index |
| `market_id` | int16 | Yes | Market identifier |
| `auth` | string | No (but needed for private data) | Auth token |

**Response fields per order:**

| Field | Type | Description |
|---|---|---|
| `order_index` | int64 | Exchange-assigned order index |
| `client_order_index` | int64 | Caller-assigned client order ID |
| `order_id` | string | String representation of order_index |
| `client_order_id` | string | String representation of client_order_index |
| `market_index` | uint8 | Market identifier |
| `owner_account_index` | int64 | Account that placed the order |
| `initial_base_amount` | string | Original order size |
| `remaining_base_amount` | string | Unfilled portion |
| `filled_base_amount` | string | Filled portion in base |
| `filled_quote_amount` | string | Filled portion in quote |
| `price` | string | Limit price |
| `is_ask` | bool | True = sell side |
| `side` | string | `"buy"` or `"sell"` |
| `type` | string | Order type enum (see above) |
| `time_in_force` | string | TIF enum (see above) |
| `reduce_only` | bool | Reduce-only flag |
| `trigger_price` | string | Trigger price for SL/TP orders |
| `order_expiry` | int64 | Expiry unix timestamp |
| `status` | string | `open`, `pending`, `in-progress`, etc. |
| `trigger_status` | string | `na`, `ready`, `mark-price`, `twap`, `parent-order` |
| `block_height` | int64 | Block when order was created |
| `timestamp` | int64 | Block timestamp |
| `created_at` | int64 | Creation time |
| `updated_at` | int64 | Last update time |
| `parent_order_index` | int64 | Parent TWAP order (if applicable) |
| `to_trigger_order_id_0` | string | Linked SL order ID |
| `to_trigger_order_id_1` | string | Linked TP order ID |
| `to_cancel_order_id_0` | string | Order to cancel on fill |
| `nonce` | int64 | Transaction nonce used |
| `integrator_fee_collector_index` | string | Integrator fee recipient |
| `integrator_maker_fee` | string | Integrator maker fee |
| `integrator_taker_fee` | string | Integrator taker fee |

**Pagination:** Uses cursor-based pagination via `next_cursor` field.

## Query: Get Order History (Inactive Orders)

```
GET https://mainnet.zklighter.elliot.ai/api/v1/accountInactiveOrders
```

**Parameters:**

| Name | Type | Required | Default | Description |
|---|---|---|---|---|
| `account_index` | int64 | Yes | — | Account identifier |
| `market_id` | int16 | No | 255 | Market filter (255 = all markets) |
| `ask_filter` | int8 | No | -1 | Filter by side (-1 = all) |
| `between_timestamps` | string | No | — | Time range filter |
| `cursor` | string | No | — | Pagination cursor |
| `limit` | int64 | Yes | — | Results per page (1-100) |
| `auth` | string | No | — | Auth token |

Response uses same Order object structure as `accountActiveOrders`. Status values
for inactive orders include: `filled`, `canceled`, `expired`.

## Transaction Log (All Order Actions)

```
GET https://explorer.elliot.ai/api/accounts/{account}/logs
```

Returns all transaction logs for an account including creates, cancels, modifies,
leverage updates, transfers. The `tx_type` field shows:
- `L2CreateOrder`
- `L2CancelAllOrders`
- `L2ModifyOrder`
- `L2UpdateLeverage`
- `L2Transfer`
- and others

Rate limited separately: 100 requests/60s; 300 requests/5m; 500 requests/10m.

## Position Management

### Get Positions

Positions are embedded in the account response:

```
GET https://mainnet.zklighter.elliot.ai/api/v1/account?by=index&value={account_index}
```

Returned as `positions` array in the `DetailedAccount` object. Each `AccountPosition`
includes: market details, position size, average entry price, position value,
unrealized PnL, realized PnL, open order count.

**Real-time positions via WebSocket:**
```
account_all_positions/{ACCOUNT_ID}     — all positions (requires auth)
account_market/{MARKET_ID}/{ACCOUNT_ID} — positions for one market (requires auth)
```

### Set Leverage

Leverage is updated via a signed transaction (`L2UpdateLeverage`, tx_type included
in transaction log). Rate limit: 40 requests/minute.

Parameters in the pubdata: `account_index`, `market_index`, `initial_margin_fraction`,
`margin_mode`.

### Close Position

No dedicated "close position" endpoint. Close by placing a market/limit order in
the opposite direction with `reduce_only=true`.

## Matching Engine

- Price-time priority: best price first; ties broken by oldest order
- Trades execute at the maker's price
- Cryptographic SNARKs verify order book matching proofs
- ZK-rollup architecture: all state settled on Ethereum L1
