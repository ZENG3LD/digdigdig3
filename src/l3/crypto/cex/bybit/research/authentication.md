# Bybit V5 API Authentication Research

**Research Date**: 2026-01-20
**API Documentation Version**: V5 (Latest)

---

## 1. Required Headers for Authenticated Requests

All private REST requests to Bybit V5 API **MUST** include the following headers:

| Header Name | Description | Example Value |
|-------------|-------------|---------------|
| `X-BAPI-API-KEY` | The API key as a string | `"XXXXXXXXXX..."` |
| `X-BAPI-SIGN` | The signature (hex or base64) | `"abc123..."` |
| `X-BAPI-TIMESTAMP` | Timestamp in milliseconds | `"1702617474601"` |
| `X-BAPI-RECV-WINDOW` | Request validity window (optional, default 5000ms) | `"5000"` |
| `Content-Type` | Request/response format (for POST) | `"application/json"` |

**Notes**:
- All headers use `X-BAPI-` prefix (not `KC-API-` like KuCoin)
- Timestamp is always in milliseconds (UTC)
- recv_window default is 5000ms (5 seconds)

## 2. Signature Algorithm

### 2.1 Signature Methods

Bybit V5 supports two signature methods:

**HMAC (System-generated keys):**
- Algorithm: **HMAC_SHA256**
- Output format: **lowercase hexadecimal string**
- Used for: API keys generated by Bybit system

**RSA (Auto-generated keys):**
- Algorithm: **RSA_SHA256**
- Output format: **Base64 string**
- Used for: Auto-generated API keys

**Recommendation**: Use HMAC_SHA256 (most common)

### 2.2 What String is Signed?

The **prehash string** format varies by HTTP method:

**For GET requests:**
```
timestamp + api_key + recv_window + queryString
```

**For POST requests:**
```
timestamp + api_key + recv_window + jsonBodyString
```

### 2.3 Component Order (EXACT)

1. **Timestamp** - milliseconds (same value as `X-BAPI-TIMESTAMP` header)
2. **API Key** - your API key string
3. **Recv Window** - request validity window in milliseconds (default 5000)
4. **Query String OR JSON Body**:
   - GET/DELETE: Query parameters (without leading `?`)
   - POST: JSON body as string

### 2.4 Query String Handling for GET Requests

**For GET requests**, include query parameters in the signature string:

**Example**:
- Full URL: `https://api.bybit.com/v5/order/realtime?category=spot&symbol=BTCUSDT`
- Query string for signature: `category=spot&symbol=BTCUSDT`
- Prehash string: `1702617474601XXXXXXXXXX5000category=spot&symbol=BTCUSDT`

**Important Notes**:
- Do NOT include the leading `?` in query string
- Parameters should be sorted alphabetically (recommended for consistency)
- Use URL-encoded values in the signature

### 2.5 Body Handling for POST Requests

**For POST requests**:
- All parameters go in the **JSON request body**
- The endpoint path in the signature **does NOT include query string**
- Example: `{"category":"spot","symbol":"BTCUSDT"}` in body

**Important**: Do NOT include extra spaces in JSON strings

### 2.6 HMAC Algorithm Steps

**Algorithm**: HMAC-SHA256

**Steps**:
1. Create prehash string: `timestamp + api_key + recv_window + (queryString OR jsonBody)`
2. Use `API Secret` as the HMAC key
3. Apply HMAC-SHA256 to the prehash string
4. Convert result to lowercase hexadecimal string
5. This is your `X-BAPI-SIGN` header value

### 2.7 Encoding

**For HMAC**: Lowercase hexadecimal string

**For RSA**: Base64 string

### 2.8 Python Example (Official)

```python
import time
import hmac
import hashlib
import json

api_key = "XXXXXXXXXX"
api_secret = "XXXXXXXXXX"
recv_window = "5000"

# Example: GET request
url = 'https://api.bybit.com/v5/order/realtime?category=spot&symbol=BTCUSDT'
timestamp = str(int(time.time() * 1000))
query_string = "category=spot&symbol=BTCUSDT"

# Prehash string for GET
str_to_sign = timestamp + api_key + recv_window + query_string

# Generate signature
signature = hmac.new(
    api_secret.encode('utf-8'),
    str_to_sign.encode('utf-8'),
    hashlib.sha256
).hexdigest()  # lowercase hex

headers = {
    "X-BAPI-API-KEY": api_key,
    "X-BAPI-SIGN": signature,
    "X-BAPI-TIMESTAMP": timestamp,
    "X-BAPI-RECV-WINDOW": recv_window
}

# Example: POST request
url = 'https://api.bybit.com/v5/order/create'
timestamp = str(int(time.time() * 1000))
json_body = {"category": "spot", "symbol": "BTCUSDT", "side": "Buy", "orderType": "Limit", "qty": "0.01", "price": "40000"}
json_body_string = json.dumps(json_body, separators=(',', ':'))  # No spaces

# Prehash string for POST
str_to_sign = timestamp + api_key + recv_window + json_body_string

signature = hmac.new(
    api_secret.encode('utf-8'),
    str_to_sign.encode('utf-8'),
    hashlib.sha256
).hexdigest()

headers = {
    "X-BAPI-API-KEY": api_key,
    "X-BAPI-SIGN": signature,
    "X-BAPI-TIMESTAMP": timestamp,
    "X-BAPI-RECV-WINDOW": recv_window,
    "Content-Type": "application/json"
}
```

## 3. Timestamp

### 3.1 Format

**Format**: Milliseconds (NOT seconds)

**Example**: `1702617474601` (milliseconds since Unix epoch)

### 3.2 Header Name

**Header**: `X-BAPI-TIMESTAMP`

### 3.3 Timezone Considerations

**Timezone**: UTC (Unix timestamp is timezone-agnostic)

### 3.4 Timestamp Validation

The server validates timestamps according to:

```
server_time - recv_window ≤ timestamp < server_time + 1000
```

**This means**:
- Your timestamp must not be older than `recv_window` milliseconds from server time
- Your timestamp must not be more than 1000ms (1 second) in the future

**Recommendation**: Use local device time synchronized with NTP

### 3.5 Important Note

The timestamp used in the **signature prehash string** **MUST** match exactly with the value in the `X-BAPI-TIMESTAMP` header.

**Correct**:
```python
timestamp = str(int(time.time() * 1000))
str_to_sign = timestamp + api_key + recv_window + query_string
headers = {
    "X-BAPI-TIMESTAMP": timestamp,  # Same timestamp
    # ...
}
```

## 4. Recv Window

### 4.1 Purpose

The `recv_window` parameter specifies how long (in milliseconds) the request is valid for.

### 4.2 Default Value

**Default**: 5000 milliseconds (5 seconds)

### 4.3 Header Name

**Header**: `X-BAPI-RECV-WINDOW`

### 4.4 Usage

**Optional**: If not specified, default 5000ms is used

**Recommended**: Use 5000ms for most cases

**Use larger value**: If experiencing network latency issues

**Example**:
```python
recv_window = "5000"  # 5 seconds
```

## 5. Differences Between Spot and Futures

### 5.1 Authentication Headers

**Conclusion**: **NO DIFFERENCE**

Both Spot and Futures APIs use the **exact same headers**:
- `X-BAPI-API-KEY`
- `X-BAPI-SIGN`
- `X-BAPI-TIMESTAMP`
- `X-BAPI-RECV-WINDOW`

### 5.2 Signature Algorithm

**Conclusion**: **NO DIFFERENCE**

Both use the same signature generation process:
- Prehash string: `timestamp + api_key + recv_window + (queryString OR jsonBody)`
- Algorithm: HMAC-SHA256
- Encoding: Lowercase hexadecimal

### 5.3 API Permissions

The only difference is in **API key permissions**:
- Spot trading permissions (order placement, cancellation, etc.)
- Derivatives trading permissions (order placement, cancellation, etc.)
- Account permissions (read balance, positions)

These are **permission settings** on the API key itself, not differences in authentication implementation.

### 5.4 Base URLs

**No difference**: Both spot and futures use the same base URL:
- **REST API**: `https://api.bybit.com`

Product differentiation is via the `category` parameter in requests.

## 7. Comparison with KuCoin Authentication

### 7.1 Key Differences

| Feature | Bybit V5 | KuCoin |
|---------|----------|---------|
| Header prefix | `X-BAPI-` | `KC-API-` |
| Signature string format | `timestamp + api_key + recv_window + params` | `timestamp + method + endpoint + body` |
| API key in signature | ✅ Yes | ❌ No |
| HTTP method in signature | ❌ No | ✅ Yes |
| Endpoint path in signature | ❌ No | ✅ Yes |
| Recv window | ✅ Yes (explicit) | ❌ No (implicit 5s check) |
| Passphrase | ❌ No | ✅ Yes (encrypted) |
| API key version | ❌ No | ✅ Yes (`KC-API-KEY-VERSION`) |
| Signature encoding | Lowercase hex | Base64 |

### 7.2 Similarities

- Both use HMAC-SHA256 for signing
- Both require timestamp in milliseconds
- Both validate timestamp against server time
- Both use API secret as HMAC key
- Both distinguish spot/futures via parameters (not auth)

### 7.3 Implementation Considerations

**Bybit is simpler**:
- No passphrase encryption required
- No API key version to manage
- Fewer headers to set
- Simpler signature string (no HTTP method, no endpoint path)

**Bybit is more explicit**:
- API key included in signature (prevents key confusion)
- Recv window explicit in signature (prevents replay attacks)

## 8. Summary

### 8.1 Key Takeaways

1. **Required Headers**: 4 headers required (`X-BAPI-API-KEY`, `X-BAPI-SIGN`, `X-BAPI-TIMESTAMP`, optional `X-BAPI-RECV-WINDOW`)
2. **Signature**: `HMAC-SHA256(timestamp + api_key + recv_window + params)` then lowercase hex
3. **No Passphrase**: Unlike KuCoin, no passphrase required
4. **Timestamp**: Milliseconds (not seconds)
5. **Recv Window**: Explicit parameter in signature (default 5000ms)
6. **Query Strings**: Include in signature for GET, use JSON body for POST
7. **Spot vs Futures**: No authentication differences, same headers and signature
8. **Simpler than KuCoin**: Fewer headers, no passphrase, no API key version
